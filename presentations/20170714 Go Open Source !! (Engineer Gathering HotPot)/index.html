もともと，ソースコードの共有は当たり前だった
The perceived value was in hardware, until Microsoft realized that software had huge potential as an industry. They were right.
The Open Source movement sought to protect the culture of sharing. The GNU Public License is one of the legal ways by which they have achieved this.
We owe a lot of the existence of Open Source today to friendly people like this guy.
And religious figures like this guy.
With all that Open Source around, being used by many of us on a daily basis, I find myself wondering why we don't contribute more. We owe so much to this movement, why aren't more people an active part of it?

オープンソースしていない理由

  • エンジニアのデバッグの方法
  • 社内のリーダー達のソフトウェアへの理解
The main reasons I've identified are the way we debug our code, and a lack of understanding software and engineering by the leadership of most of the companies we work at.

落ち着けば大丈夫

Getting to the root of the problem, is quite literally an exercise in digging. At our company, I really push for people to go through these steps one by one, and not give up until the root cause of the problem has been identified. This can be scary, but it is totally worth it.

バグを報告しよう！

  • バグを報告する責任があります
  • アナタは間違えているかもしれない
  • 間違っていると、ドキュメントの不備やAPIデザインの不具合が発見される可能性があります
Authors want to know about your problems so they can improve the library for you, themselves and others. Keep in mind that library authors can often have tunnel vision about the product they're working on. That can make your feedback invaluable.
でも納期は迫っている！
I understand you would be worried about your deadlines. Reporting a bug tells you nothing about when you will be able to deliver your blocked feature.
自分で直して，プルリクエストを出そう．
- Fancy developers are just people. - Have confidence in your own abilities. - Jump into the project with confidence, debug your butt off, you will succeed. - Because existing contributors will often gladly help you when you get stuck (they will see a potential new contributor in you).

オープンソースのバグを積極的に修正すると(1/2)

  • 適切なレビューを受ける事が出来るため、修正したプログラムが確実に動作する
  • もし修正したプログラムがマージされなくても，あなたはそのプログラムを使うことは出来ます(だから納期には間に合う)
  • あなたにとって大切なソフトウェアのコミュニティメンバーになれます
- Ensures the fix works, as it will receive proper review. - Will give you a public branch to continue to work from (deadline safe). - Become a member of a community around software in which you have a vested interest. That last point is especially important to me. As a community, if you can save a ton of people the effort of debugging the same issue, by spending 4 hours on a pull request yourself, you will save a lot of man-hours. I call that pretty productive. "Work smart, not hard", right?
When we acknowledge that our interests align, this should be the obvious course. Our bosses may not always recognise it, but in an open source world, all developers are co-workers. We're in this together.

オープンソースのバグを積極的に修正すると(2/2)

  • あなたのデバックスキルが向上します
  • そのソフトウェアのパフォーマンスの
    引き出し方に詳しくなれます
  • あなたがバクを修正することで
    世界中のエンジニアの役に立ちます
- Improves debugging of your own software. - Improves understanding of performance bottlenecks. - Prevents others from wasting time. That last point is especially important to me. As a community, if you can save a ton of people the effort of debugging the same issue, by spending 4 hours on a pull request yourself, you will save a lot of man-hours. I call that pretty productive. "Work smart, not hard", right?
When we acknowledge that our interests align, this should be the obvious course. Our bosses may not always recognise it, but in an open source world, all developers are co-workers. We're in this together.

    Some tips

    • 多くのプロジェクトにはコントリビュートの方法が書かれています．読みましょう！
    • 多くのプロジェクトには行動規範があります．それに従い，行動する事が大切です
    • たとえ他の人の行動がふさわしくないと思った時でも、あなたの寛大さは重要です
    • 相手のプロジェクトのスタイルを壊してはいけません
    - Code of Conduct: It protects you and others working on the project. - Code style: When you visit a friend’s house, you don’t start redecorating their home, right? Treat a project’s code the same, some people are spending their whole days in there.

    コントリビューションしやすい場所

    • Documentation
    • Unit tests
    • Benchmarks
    If you're a bit scared to get into a particular project and want to familiarize yourself with it a bit more first, these are 3 ways to add value to the project without having mastered it. Users are often better at writing documentation than the code authors themselves. And in the process, you will understand the code base more and more, and will be able to start contributing code.

    Don't know where to start?

    Help wanted!

    • Node.js and Electron modules that have compiled parts.
    • PhoneGap plugins which share both JS and Java/ObjC.
    If you have some time to spare and want to help out, there are many libraries that mix languages which can make it hard to find people to contribute. The mix of languages can scare people away from contributing. But if you're a polyglot and are comfortable around multiple environments, this category of libraries could really use your help.
